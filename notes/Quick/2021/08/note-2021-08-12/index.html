提供一套操作 git repositories 接口的服务，居然是 Go 实现的。

现在存储 repository 的发送是 Hashed storage。差点我还以为 repository 丢失了。

Hashed storage is different to legacy storage where a project is stored based on:

  • The project’s URL.
  • The folder structure where the repository is stored on disk.

gitlab-workhorse

之前 gitlab 用的 http 服务是 unicorn，因为超时问题，对大文件上传下载，git http 的支持很麻烦。所以重新用 go 实现了一个 gitlab-http-git-server 。后来发展成为gitlab 的前置路由，所有请求经过 gitlab-workhorse 再分发给其他服务。名字也改了，对应独角兽改成驮马。

https://about.gitlab.com/blog/2016/04/12/a-brief-history-of-gitlab-workhorse/

严格安装文档进行恢复 Back up and restore GitLab | GitLab。

结果依然遇到问题 ActiveRecord::IrreversibleOrderError，主要结合 @superbiche 的回复，重新恢复才解决问题。

  • gitlab-ctl stop
  • 只启动 postgresql gitlab-ctl start postgresql
  • 切换到 gitlab-psql su - gitlab-psql
  • 登陆 postgresql /opt/gitlab/embedded/bin/psql -h /var/opt/gitlab/postgresql -d postgres 任意一个非 gitlabhq-production 的数据库
  • drop any connection left in Postgres pod as postgres user select pg_terminate_backend(pid) from pg_stat_activity where datname='gitlabhq_production';
  • drop gitlabhq_production db drop database gitlabhq_production;
  • 关键步骤 grant superuser privileges to gitlab user ALTER USER gitlab WITH SUPERUSER;
  • gitlab-rake db:reset

然后就可以重新继续恢复流程，见 Back up and restore GitLab | GitLab。

  • 停用服务比文档多一些 I left {alertmanager, nginx, sidekiq, sshd, unicorn} stopped. 也有可能是因为这个起作用。

最后恢复 gitlab 的权限 ALTER USER gitlab WITH NOSUPERUSER;

恢复数据库期间会一开始出现大量 ERROR 是正常的，见 Error restoring PostgreSQL (#4851) · Issues · GitLab.org / omnibus-gitlab · GitLab
