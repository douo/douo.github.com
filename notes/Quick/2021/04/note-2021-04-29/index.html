前端/客户端技术线条更新初步计划–声明式 UI

目标

传统的命令式 UI，手动更新视图会增加出现错误的可能性。比如一条数据在多个位置呈现，很容易忘记更新其中一个显示数据的视图，或者数据发生不同的状态变化，需要手动更新视图的不同 UI 元素，导致代码量膨胀。随着 UI 逻辑日益复杂，软件维护的复杂度也会成指数增加。

在过去的几年中，GUI 开发已开始转向声明式 UI 模型，该模型极大地简化了与构建和更新用户界面相关的工程设计，简单说该技术的工作原理就是数据发生变化时重新生成整个屏幕，然后仅应用（通过算法优化后）必要的更改。这种方法避免了手动更新有状态视图层次结构的复杂性。

如今，客户端的声明式 UI 框架面世也有一段时间，相关 Toolkit 逐渐成熟、IDE 也能有较好支持。尽管投入生产可能还有一些限制，比如 SwiftUI 只支持 iOS 13.0+，Jetpack Compose 还处于 1.0beta。但我认为声明式 UI 是一段时间内行业的发展趋势，与降低 UI 开发成本、降低维护 UI 的复杂度相比，这些暂时的短板不应成为阻碍。

所以我选择的技术线条更新目标是，掌握声明式 UI 的编程模型，掌握各自平台上的声明式 UI 框架的最佳实践。

学习路线

  1. 了解声明式 UI
  2. 掌握基本函数式编程范式及概念
    • 掌握所用语言对函数式编程的支持
    • 了解函数式编程的概念，比如掌握响应函数，需要了解幂等、不变性、副作用等概念
    • 掌握替换命令式 UI 常用的面向对象编程技巧，比如用组合代替继承、高阶函数代替多态等
  3. 掌握各自平台上的声明式 UI 框架
    • iOS: SwiftUI
    • Android: Jetpack Compose
    • Web: React/Vue
    • 掌握数据更新、事件处理、常用内置组件等
  4. 整合响应式数据与状态管理
    • iOS: combine、rxswift 等
    • Android: livedata、flow 等
    • Web: redux、vuex 等
    • 了解单一可信源、单向数据流、状态提升等概念
  5. 了解声明式 UI 的实现原理及局限性
  6. 实践

实践与考核

实践目标是基于声明式 UI 开发一款简单列表-详情的 App/WebApp。App 通过网络请求数据并展示，同时具有一定交互操作，和本地数据存储。

考核的点如下：

  • 实现相对复杂的布局
  • 一般化 LCE（加载-内容-出错）模式
  • 列表项插入、删除及更新
  • 数据 immutable
  • 用函数式风格的代码操作列表
  • 声明函数能正确实现为纯函数
  • 能处理可能带有副作用的情景，比如点赞、收藏等

进阶考核：

  • 用声明式 UI 实现动画和手势

人员与时间

不影响开发任务的前提下，保守需要 3 个月的学习时间。1、2两个月的实践时间。
